# CONFLICT RESOLUTION NOTES:
# --------------------------
# 1. PyTorch vs JAX CUDA Bindings:
#    - Both frameworks require compatible CUDA/cuDNN versions to prevent ABI mismatches.
#    - Resolution: Pin torch==2.2.0 and jax==0.4.25 with jaxlib==0.4.25, both targeting CUDA 12.1.
#      This ensures shared cuDNN 8.9 and NCCL 2.19 compatibility for multi-GPU workloads.
#      Using 'cuda-python' as common binding layer prevents duplicate CUDA runtime loading.
#
# 2. Ray Memory Management:
#    - Ray's Plasma store can conflict with PyTorch's CUDA memory allocator.
#    - Resolution: Pin ray==2.9.0 with explicit plasma configuration. Use PYTORCH_CUDA_ALLOC_CONF
#      environment variable to prevent fragmentation during shadow deployments.
#
# 3. PyO3 GIL Deadlock Prevention:
#    - Rust FFI calls from Python can deadlock if GIL is held during async operations.
#    - Resolution: pyo3==0.20.0 with abi3-py310 feature. All async boundaries use
#      pyo3-asyncio with tokio-runtime to ensure proper GIL release across FFI.
#
# 4. Qiskit vs PennyLane Quantum Backends:
#    - Both provide quantum circuit execution but may load conflicting native libraries.
#    - Resolution: qiskit==1.0.0 and pennylane==0.35.0 use isolated backend loaders.
#      Qiskit uses Aer simulator; PennyLane uses default.qubit. No shared library conflicts.
#
# 5. Transformers vs vLLM Model Loading:
#    - Both load HuggingFace models but vLLM uses custom CUDA kernels.
#    - Resolution: transformers==4.38.0 and vllm==0.3.0 share same tokenizers==0.15.0.
#      vLLM's PagedAttention kernels are isolated from transformers' attention implementations.
#
# 6. NumPy ABI Compatibility:
#    - Critical for PyO3 numpy integration and all scientific computing stack.
#    - Resolution: numpy==1.26.0 with pinned C-API version for stable FFI boundaries.
#
